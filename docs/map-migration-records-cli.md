# map-migration-records CLI

`vdbg map-migration-records` validates how a planned pgvector schema can drive future full-record Milvus-to-pgvector migration.

It consumes only the local JSON artifact generated by `vdbg plan-pgvector-schema` and writes a deterministic record mapping report.

## Position in the migration flow

```text
inspect-milvus
  -> plan-pgvector-schema
  -> compare-schema-plans
  -> apply-pgvector-schema
  -> inspect-pgvector-schema
  -> compare-applied-schema
  -> map-migration-records
  -> migrate / future full-record migration execution
```

## Command

```bash
go run ./cmd/vdbg map-migration-records \
  --schema-plan /tmp/vdb-guardian-pgvector-schema-plan.json \
  --output /tmp/vdb-guardian-record-mapping.json
```

Without `--output`, the JSON report is printed to stdout:

```bash
go run ./cmd/vdbg map-migration-records \
  --schema-plan /tmp/vdb-guardian-pgvector-schema-plan.json
```

## Safety boundary

The command:

- does not connect to Milvus;
- does not connect to PostgreSQL;
- does not execute DDL or DML;
- does not read or migrate row payloads;
- does not require or print connection URLs;
- writes `--output` artifacts with `0600` permissions.

## Mapping rules

The report classifies planned columns as:

- `primary_key`: planned primary key column;
- `vector`: dense pgvector column whose target type starts with `vector`;
- `scalar`: regular scalar source field to target column;
- `dynamic_metadata`: `_milvus_dynamic` metadata JSON column;
- `partition_metadata`: `_milvus_partition` metadata column.

Blocking issues include:

- missing primary key mapping;
- missing vector mapping;
- multiple primary key mappings;
- multiple vector mappings;
- unsupported planned columns.

When blocking issues exist, the command writes the report first and then returns non-zero.

## Example report shape

```json
{
  "schema_version": "v1",
  "schema_plan": "/tmp/vdb-guardian-pgvector-schema-plan.json",
  "status": "pass",
  "mappings": [
    {
      "source_collection": "items",
      "target_schema": "public",
      "target_table": "items",
      "primary_key": {
        "kind": "primary_key",
        "source_field": "id",
        "target_column": "id",
        "target_type": "text",
        "nullable": false,
        "support_level": "supported"
      },
      "vector": {
        "kind": "vector",
        "source_field": "embedding",
        "target_column": "embedding",
        "target_type": "vector(1536)",
        "nullable": false,
        "support_level": "supported"
      }
    }
  ],
  "summary": {
    "collection_count": 1,
    "scalar_mapping_count": 0,
    "dynamic_metadata_mapping_count": 0,
    "partition_metadata_mapping_count": 0,
    "issue_count": 0,
    "blocking_issue_count": 0
  }
}
```

## Current limitations

This phase only validates the mapping shape. It does not yet wire scalar, dynamic-field, or partition payload extraction into the real `vdbg migrate` writer. That execution work belongs to the next full-record migration phase.
